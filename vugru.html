VuGru

Photo review and approval platform

Photographers lose edits and delay projects because client feedback arrives fragmented across emails, texts, and messages. VuGru centralizes photo review, feedback, and approval into a single workflow so projects close cleanly.

Course 1 — Product Design
Product Manager

Defined design intent, conducted user research, developed personas and Jobs to be Done, and translated findings into a prioritized epic structure with outcome-anchored user stories.

Course 2 — Coding with GPT
Engineering Team Lead

Led the team through full implementation — decomposing user stories into view, controller, and model logic, running two rounds of usability testing, and iterating based on observed behavior.

The Problem

Photographers working with clients face a consistent breakdown: feedback arrives across three or four channels simultaneously, revisions get missed, and projects drag on because neither side has a clear view of what's been addressed and what hasn't. The root cause isn't communication volume — it's the absence of a shared source of truth that ties each piece of feedback to a specific photo and tracks its status through to resolution.

The guiding design principle throughout both product and code: clarity over complexity, intuition over instruction.

Personas and Jobs to be Done

P
Paahil
The Photographer

"When I send draft albums, I need feedback captured in one place so that I don't lose edits across multiple channels and can finish revisions on time."

Save time, reduce mistakes, move projects to completion smoothly.
C
The Client
Album Reviewer

"When I review my photos, I want to give clear feedback on the right image and know revisions are tracked, so I feel confident approving the final album."

Give feedback simply, track progress transparently, close projects without friction.

Three design implications followed directly from the personas: feedback must be photo-specific, interactions must be discoverable without instructions, and status visibility must reduce uncertainty for both roles. Every subsequent product and engineering decision was evaluated against these three constraints.

What Was Built

The final product implements fourteen user stories across the core workflow: album upload, client notification, photo review, photo-specific commenting, version tracking across revision cycles, role-based access for photographers and clients, and a project status system that surfaces where each engagement stands.

Upload
Photographer uploads a draft album; client receives a notification without searching through emails.
Review
Client opens the album, views photos in an intuitive scrollable grid, enlarges on click.
Feedback
Client leaves comments tied directly to specific photos. No ambiguity about which image each comment applies to.
Revisions
Photographer uploads revised photos labeled as new versions. Client sees what changed.
Curation
Client marks photos for the final album without needing to leave a comment for each selection.
Approval
Client approves the final album. Project closes. Payment moves forward.

How It Was Built

View layer
Grid-based photo gallery, enlarged photo modal, inline comment sections per photo, clear button hierarchy for approval actions.
Controller logic
Photo selection and enlargement, comment creation tied to photo IDs, like and Add-to-Album state toggling, status transitions across the workflow.
State management
Photo objects with metadata (id, version, status), comment objects scoped to photo IDs, in-memory state that reflects workflow progression.
Build decision framework

If a component shaped user behavior, we built it from scratch. If it supported the workflow without defining it, we considered external solutions. Authentication was the only third-party integration — a non-differentiating capability that didn't touch the core feedback workflow. Every view, controller, and model for photo display, commenting, and approval was written ourselves to maintain tight alignment between design intent and implementation behavior.

Testing and Iteration

Two rounds of usability testing shaped the final implementation. The first round surfaced layout assumptions that didn't hold under real use — comments were relocated, button hierarchy was simplified, gallery behavior was adjusted. The second tested the end-to-end flow from upload through approval and validated the revised interactions.

End-to-end tests were generated using Playwright via Cursor. Early test generation produced 66 tests, most of which lacked the contextual preconditions that real user flows require — tests tried to add comments when no albums existed, for example. Refining the scenarios to include proper "given/when/then" context produced a test suite that actually reflected user behavior and passed cleanly.

The debugging philosophy that emerged: treat confusion as a design bug, not just a code bug. When something wasn't working, the first question was whether the UI state matched the user's mental model — not just whether the code was syntactically correct.

What I Learned

Design intent has to be specific enough to be testable. "Intuitive" is not a behavioral definition of done. "A client can leave feedback without ambiguity about which image it applies to" is.

Building the core workflow from scratch was the right call. It kept iteration fast — small changes to layout or interaction could be implemented and tested without working around external frameworks.

Test quality depends on context, not count. 66 context-free tests were less useful than 20 tests grounded in real user flows. The discipline of writing "given/when/then" properly applies to testing the same way it applies to user stories.

Shipping changes what you know. The gap between a prototype that looks right and a deployed product that behaves right is where the real product thinking happens.
